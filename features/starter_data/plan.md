## Feature Summary

Build the `customer_dim` mart that consolidates customer state from
`source.customers` with daily refresh. Downstream consumers: Looker
customer-360 dashboard, the marketing reverse-ETL job (Hightouch →
Marketo).

## Task Breakdown (Ordered)

1. **Add stripe source declaration** — `models/staging/stripe/_sources.yml`
   declares `source('stripe', 'customers')` + freshness contract (warn 12h,
   error 24h).
   - Tests: source freshness configured.

2. **Build `stg_stripe__customers.sql`** — staging model.
   - Renames + type casts
   - PII columns (`email`, `phone`, `billing_address`) wrapped in `mask_pii()`
   - Tagged in `_stg_stripe__customers.yml`
   - Tests: `unique` + `not_null` on `customer_id`; PII tag coverage CI gate

3. **Build `int_customers__active_status.sql`** — intermediate.
   - Joins `stg_stripe__customers` with subscription state (separate intermediate)
   - Computes `is_active` flag
   - Materialization: ephemeral

4. **Build `dim_customers.sql`** — the mart.
   - Reads from intermediate models only
   - Materialization: table (small) OR incremental on `dim_loaded_at`
     (requires ADR if incremental)
   - PII columns re-tagged at mart level
   - Tests: PK unique + not_null; FK relationships; row count drift (warn)

5. **Custom singular tests** — `tests/dim_customers_*.sql`
   - `dim_customers_email_lowercase.sql`
   - `dim_customers_dim_loaded_at_monotonic.sql`

6. **Exposures** — `models/marts/_exposures.yml`
   - `customer_360_dashboard` (Looker) — owner: analytics-eng
   - `marketing_reverse_etl` (Hightouch) — owner: martech

7. **Freshness check** — post-run script that queries
   `max(dim_loaded_at)` and emits the staleness metric. Wired to oncall.

8. **PII masking validation** — synthetic-customer canary in CI that
   queries `dim_customers` in the ci environment and asserts masked
   format on all PII columns.

9. **Backfill instructions** — README block: `dbt build --select
   customer_dim+ --vars '{start_date, end_date}'` with idempotency
   guarantee documented.

10. **Lineage tool wiring** — TBD which tool (see NFRs file). Confirm
    `manifest.json` is ingested by the chosen tool.

## Risks + Open Questions

- **Source schema drift** — Stripe occasionally renames columns silently;
  freshness alert won't catch this. Mitigation: schema test on column
  presence in staging.
- **Backfill cost** — a full-history backfill is TBD credits. May need
  approval before running in prod.
- **PII column inventory** — confirm with security: are there additional
  fields beyond email/phone/address/tax_id that count as PII?
- **Lineage tool** — open question; needs decision before this ships to
  prod.
